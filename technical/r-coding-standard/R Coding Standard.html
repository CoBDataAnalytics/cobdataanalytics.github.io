Chapter 01

Introduction to R & RStudio

1.1 What is R?

R is a free, open-source programming language and software environment specifically designed for statistical computing and graphics. It is highly extensible and has become the industry standard for data analysis, providing a vast ecosystem of packages maintained by the global data science community.

1.2 What is RStudio?

RStudio is an Integrated Development Environment (IDE) for R. It provides a user-friendly interface that includes a code editor, console, plotting window, and environment viewer. RStudio makes writing, debugging, and executing R code significantly easier and more efficient than using base R alone.

1.3 Why R in D&A?

The Data Analytics & Insights team leverages R to ensure our analytical outputs are fully reproducible and transparent. By combining R with the tidyverse meta-package, we can perform complex data manipulation (dplyr) and generate publication-ready visualizations (ggplot2) in a structured format that can be easily shared, audited, and replicated by others.

1.4 Getting Started with R and RStudio

To get started with R and RStudio, follow these steps:

  1. Install R from the Comprehensive R Archive Network (CRAN). As R is not in the suite of software tools, you will need to request installation permission through the request portal.
  2. Install RStudio from the official RStudio website.
  3. Launch RStudio and familiarize yourself with the interface.
Chapter 02

Purpose & Principles

Audience

This document is written as a manual for anyone working in the D&A team, but also as a guide for anyone who would like to write clean and clear code that is meant to be shared.

2.1 Purpose

Maintaining reproducibility and transparency is a core value of the Data Analytics & Insights team, and an R style guide can help us achieve that goal. Additionally, adhering to the basic rules in this style guide will improve our ability to share, maintain, and extend our analytics when working with R.

2.2 Principles

  • We take a disciplined and practical approach to writing code.
  • We regularly check-in code to Azure DevOps.
  • We believe consistency in style is very important.
  • We demonstrate intent explicitly in code, via clear structure and comments where needed.
Chapter 03

Core Rules

Standard

D&A follows the conventions outlined in the tidyverse style guide. This is assisted by the R packages styler and lintr which support the tidyverse style guide.

To ensure compliance across the team, all R scripts should be processed through the Styler add-in within RStudio before finalising pull requests. This automated approach removes the friction of manual formatting and guarantees that all code contributed to the repository adheres to the exact same spatial and syntactical standards.

Chapter 04

Additional Rules

Supplement

On top of the tidyverse style guide, there are additional rules put in place for R coding within D&A. These are outlined below.

4.1 General Practices

  • Check code into Azure DevOps early and often.
  • Under Code Options, enable "Insert spaces for Tab" with a Tab width of 2.
  • No trailing whitespace.
  • Try to only comment on things that aren't obvious about the script (e.g., function is used, etc.), that explains the reasoning behind your code.

4.2 R Project Files

When working within a repository for R, always create an RStudio Project file. An R project file, typically denoted by a .Rproj extension, helps maintain a clean and structured workflow by encapsulating your R scripts, data, and associated files within a single directory.

Using an R Project file will ensure that the working directory will always be set to the correct location, which will enable the usage of relative path references. You can also set specific project related settings to ensure consistency across your project.

4.3 Documentation

At the beginning of your R script, document the name of the script, purpose of the script and the developers involved. This brief documentation will provide useful information to others about the script, and will also assist you in the future when revisiting your scripts.

################################################################
## Name: Addition Of Two Numbers
## Purpose: This script adds two numbers specified by the user
## Lead Developer: Shaq O'Neill (Senior Analyst - D&A)
## Additional Developers (optional): 
##    Steph Curry (Senior Data Analyst - D&A)
##    Lebron James (Data Analyst - D&A)
################################################################

4.4 Optimise Installing and Loading Packages

Although RStudio now has automatic checking for missing required packages, it is handy to incorporate installing and loading packages at the start of your script rather than multiple library calls. It should be noted that packages that are not available on CRAN are unable to be installed through the below script.

# Package names
packages <- c("dplyr", "ggplot2", "hrbrthemes")

# Install packages not yet installed
installed_packages <- packages %in% rownames(installed.packages())
if (any(installed_packages == FALSE)) {
  install.packages(packages[!installed_packages])
}

# Packages loading
invisible(lapply(packages, library, character.only = TRUE))

4.5 Other Conventions

Variable and Function Names: Use snake_case for variable and function names. Be descriptive in your naming to enhance readability. Avoid overly short or cryptic names. When formatting a value as a count or percentage, append this to the end of the variable (e.g., separation_count, separation_percentage).

Consistency: Maintain consistent naming conventions throughout your codebase. If you start with snake case, stick to it. When defining column names in data frames, follow the same snake case pattern (e.g., codes_count, hospital_identifier).

Commenting: Commenting what your code does gives another reader the opportunity check that your code does what you intend. Include why you are coding, i.e., if its used in text later, or if you are calculating a numerator for a value. Large comment chunks are fine to break up your code.

Avoid Index-based Access: Where possible, avoid index-based access when working with dataframes (e.g., df[,1] <- NA). Index-based access may run into issues in the future if the data structure changes. If possible, refer to the column names to safeguard against this (e.g., df[, "my_column"] <- NA).

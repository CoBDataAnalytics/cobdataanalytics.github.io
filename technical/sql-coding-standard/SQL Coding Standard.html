Chapter 01

Overview & Purpose

Scope

This style guide sets forth guidelines for production SQL code deployed by the Data Analytics & Insights (D&A) team. It is tailored for Azure SQL (T-SQL), but the principles can be applied across other SQL dialects.

1.1 Purpose

Maintaining reproducibility and transparency is a core value of the D&A team, and a SQL style guide helps us achieve that goal. Adhering to these rules will improve our ability to share, maintain, and extend our analytics when working with SQL.

1.2 Audience & Adoption

This document is a manual for anyone in the D&A team, particularly those developing, reviewing, or maintaining SQL code for deployment in production environments (including stored procedures, data pipelines, and reporting tools).

  • All new SQL code is expected to adhere to these standards.
  • Existing code should be updated to align where practical, especially when modifications are being made.
  • Consistent implementation - especially in structure and naming - is important, as partial adoption reduces effectiveness.
Chapter 02

Principles & Formatting

2.1 Core Principles

  • We take a disciplined and practical approach to writing code.
  • We believe consistency in style is very important.
  • We demonstrate intent explicitly in code, via clear structure and comments where needed.

2.2 Indentation & Spacing

Consistent indentation improves readability and reduces errors during maintenance. We use 2 spaces per indent (not tabs).

  • When using VSCode, set "editor.tabSize": 2, "editor.insertSpaces": true, and "editor.detectIndentation": false.
  • Each major clause (SELECT, FROM, WHERE, JOIN) should appear on a new line, with subordinate elements indented.
  • No trailing whitespace.

2.3 Capitalization & Keywords

Always capitalize SQL keywords (e.g., SELECT, AS, INNER JOIN). Never name an object with a SQL Server reserved word.

2.4 Documentation & Comments

All SQL code should include clear documentation explaining intent, assumptions, and non-obvious logic. Use inline comments sparingly; prefer block comments at the start of scripts.

  • Comments should go near the top of your query, or at least near the closest SELECT.
  • Only comment on things that aren't obvious (e.g., why a particular Council ID is hardcoded).
Chapter 03

Object Naming Standards

3.1 General Naming Rules

Object names will be English words conveying meaning about what the object represents. Avoid numbers, proprietary acronyms, or product-specific names whose meaning might change.

  • Use the singular form (e.g., Property, not Properties).
  • Use 'Camel capitalization' (PascalCase) for tables and views (e.g., WasteCollection, PlanningApplication).

3.2 Variables & Aliases in Queries

Variable names and column aliases should be underscore-separated and capitalised (SCREAMING_SNAKE_CASE). Do not use single-letter names; be as descriptive as possible.

-- GOOD: Descriptive aliases
SELECT 
  PROP.PROPERTY_URN AS PROPERTY_URN,
  RES.RESIDENT_ID AS RESIDENT_ID,
  COUNT(WASTE.COLLECTION_ID) AS COLLECTION_COUNT
-- BAD: Single letters and camelCase aliases
SELECT 
  PROP.PROPERTY_URN,
  RES.RESIDENT_ID AS rId,
  COUNT(WASTE.COLLECTION_ID) AS c

3.3 Table Aliases

Always provide aliases for your tables when joining. Do not use single-letter table names.

-- GOOD
FROM Council.Property AS PROP
INNER JOIN Council.Resident AS RES 
  ON PROP.PROPERTY_URN = RES.PROPERTY_URN
-- BAD: Missing AS keyword and single letter alias
FROM Council.Property P
INNER JOIN Council.Resident R 
  ON P.PROPERTY_URN = R.PROPERTY_URN

3.4 Column Categorization

Categorize columns logically to maintain consistency across the council's data estate:

  • Identifiers: PropertyId, ApplicationId
  • Descriptive fields: PropertyName, WardName
  • Dates/timestamps: CollectionDate, ApprovalDate
  • Flags: IsActiveFlag, IsDemolished
Chapter 04

Query Structure & Syntax

4.1 SELECT Statements

SELECT goes on its own line. Align all columns to the first column. Always rename aggregates and function-wrapped columns. Always select with table aliases if you have joins, and rename all columns when doing so.

SELECT
  PROP.PROPERTY_URN AS PROPERTY_URN,
  RES.LAST_NAME AS LAST_NAME,
  SUM(CTAX.AMOUNT_DUE) AS TOTAL_RATES_DUE
FROM Council.Property AS PROP
INNER JOIN Council.Resident AS RES 
  ON PROP.PROPERTY_URN = RES.PROPERTY_URN
INNER JOIN Council.CouncilTax AS CTAX 
  ON PROP.PROPERTY_URN = CTAX.PROPERTY_URN

4.2 FROM & JOIN Statements

Only one table should be in the FROM clause; never use legacy `FROM`-joins (comma separated). Explicitly use INNER JOIN rather than just JOIN. Begin with INNER JOINs, then list LEFT JOINs - do not intermingle them unless absolutely necessary.

The ON keyword goes on a new indented line. Additional filters go on a double-indented line beginning with AND.

FROM Council.Property AS PROP
INNER JOIN Council.CouncilTax AS CTAX 
  ON PROP.PROPERTY_URN = CTAX.PROPERTY_URN
    AND CTAX.FINANCIAL_YEAR = 2024
INNER JOIN Council.Ward AS WARD 
  ON PROP.WARD_CODE = WARD.WARD_CODE
LEFT JOIN Council.DemolitionOrder AS DEMO 
  ON PROP.PROPERTY_URN = DEMO.PROPERTY_URN

4.3 WHERE & GROUP BY

Multiple WHERE clauses should go on different lines, beginning with the SQL operator. Align GROUP BY columns to the first column. In Azure SQL, you can utilise named variables from your SELECT statement in your GROUP BY.

WHERE
  WARD.COUNCIL_AREA = 'NORTH'
  AND PROP.PROPERTY_TYPE = 'RESIDENTIAL'
GROUP BY
  PROP.PROPERTY_URN,
  TOTAL_RATES_DUE

4.4 ORDER BY & CASE

Be explicit in the direction of ordering (ASC/DESC). For CASE statements, align each WHEN and THEN on a new indented row.

ORDER BY
  WARD.WARD_NAME ASC,
  TOTAL_RATES_DUE DESC

-- CASE Example
CASE 
  WHEN PROP.PROPERTY_TYPE = 'RESIDENTIAL' THEN 'DOMESTIC'
  WHEN PROP.PROPERTY_TYPE = 'COMMERCIAL' THEN 'BUSINESS'
  ELSE 'OTHER'
END AS PROPERTY_CATEGORY

4.5 Window Functions

Long window functions should be split across multiple lines: one for PARTITION BY, ORDER BY, and frame clauses. Multiple partition keys should be one-per-line. Order should always be explicit. All window functions must be aliased.

ROW_NUMBER() OVER (
  PARTITION BY 
    PROP.PROPERTY_URN,
    CTAX.FINANCIAL_YEAR
  ORDER BY CTAX.BILL_DATE DESC
  ROWS UNBOUNDED PRECEDING
) AS LATEST_BILL_RANK

4.6 Common Table Expressions (CTEs)

A CTE is a named subquery defined in a WITH clause, acting as a temporary view. Always use CTEs over inlined subqueries to improve readability. The body of a CTE must be one indent further than the WITH keyword. Close them on a new line.

WITH RATES_PER_PROPERTY AS (
  SELECT
    PROPERTY_URN,
    SUM(AMOUNT_DUE) AS TOTAL_RATES_DUE
  FROM Council.CouncilTax
  GROUP BY PROPERTY_URN
), 
PROPERTY_DETAILS AS (
  SELECT
    PROP.PROPERTY_URN AS PROPERTY_URN,
    PROP.ADDRESS_LINE_1 AS ADDRESS_LINE_1,
    RATES.TOTAL_RATES_DUE AS TOTAL_RATES_DUE
  FROM Council.Property AS PROP
  INNER JOIN RATES_PER_PROPERTY AS RATES 
    ON PROP.PROPERTY_URN = RATES.PROPERTY_URN
)
SELECT * FROM PROPERTY_DETAILS
Chapter 05

Best Practices & Data Types

5.1 Query Performance & Clarity

Write queries that are efficient, maintainable, and predictable.

  • Avoid SELECT *; instead, explicitly specify required columns.
  • Use proper joins with clear conditions.
  • Ensure filters (WHERE) are applied as early as possible (e.g., inside CTEs before joining to large tables).
  • Avoid unnecessary subqueries when standard joins or CTEs improve clarity.

5.2 Data Types & Integrity

Use appropriate and consistent data types to ensure data integrity and performance across the council's Azure SQL databases.

  • Avoid overly generic types such as NVARCHAR(MAX) unless strictly necessary (e.g., holding JSON payloads or long text descriptions).
  • Prefer explicit sizing and standardized types across similar fields (e.g., all IDs use INT or BIGINT, timestamps use DATETIME2).

5.3 Version Control

Check code into Azure DevOps (GitHub) early and often. Treat SQL scripts with the same rigor as application code - ensure every deployable script is reviewed and tracked in version control before it is executed against a production environment.
